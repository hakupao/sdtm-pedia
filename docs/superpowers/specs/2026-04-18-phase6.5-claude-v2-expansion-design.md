## 1. 背景与目标

### 1.1 触发链

1. v1 上传 (192K) 显示 12% capacity, 与预期 95% 偏差 8 倍
2. capacity_research 修订: paid 套餐 RAG 自动扩容 ~3-4M
3. v1 在 T1-T8 全 PASS, 但 T3/T6/T7 出现"间接重建/兜底", 损失原文级精确度
4. 用户挑战: "为何不直接占到 90%?" (见 §1.4)

### 1.2 目标 (中庸 / 50%)

**主目标**:
- v2.6 终态 token ~1.5M (RAG 容量 ~50%)
- 在 v1 T1-T8 上不衰减, T3/T6/T7 升级为原文精确
- T9-T20 新增题集全 PASS (覆盖六批新增内容)
- 建立 RAG 质量 vs 规模曲线 (6 个数据点: v1 + v2.1-v2.5 + v2.6)

**辅目标**:
- 留 50% capacity buffer 给: 对话上下文、Phase 7 知识图谱注入、CDISC 标准更新、未来扩到 70%
- 输出 RAG 衰减观察报告作为 Phase 7 设计输入
- 失败回退路径完整 (任何阶段衰减 ≥2 题立即停)

### 1.3 非目标

- 不动 knowledge_base/ 任何文件 (P5)
- 不替换 v1 Project, 不删 output_v1_baseline/
- v2 不追求 90% 满载 (留 buffer + 边际收益不足, 见 §1.4)
- terminology Term 值不全展开 (1944K 中, v2 取 top 500 codelist 即可, 全展开留 Phase 6.5+ 评估)

### 1.4 为什么不 90% (取舍记录)

| 论点 | 90% 派 | 50% 派 (本设计采纳) |
|------|-------|------|
| 已付费容量 | "12% 浪费 88%" | RAG 容量不是"必须用满"的资源, 留 buffer 是质量保障 |
| 精确度提升 | "全展开后无需边界模板" | 70%+ 时低密度 terminology Term 值占比过大, 价值密度急降 |
| RAG 衰减 | "未实测, 不假设衰减" | 未实测就更要保守起步, 建曲线再决定终点 |
| 维护成本 | "全文上传, 无需脚本" | 仍需脚本: terminology 1944K 全上传超 RAG 索引能力, 必须分组+部分压缩 |
| 不可逆性 | "可删可重传" | reindex 等待 + 用户操作疲劳, 渐进上传是稳妥做法 |
| 边际收益 | "100% 信息密度" | 95% 查询命中前 30-50% 内容, 后 40% 是长尾 |

**结论**: 中庸 50% 是"高 ROI 区段的上界 + 可控风险的下界", 70% 留作下轮目标。

### 2.4 批 3: examples 高频域数据表 (+50-100K)

**圈定算法** (B方案 + 用户高频):

**必选 (用户 20 域)**:
DM, SE, DS, BS, BE, MI, GF, PR, CM, EX, TU, TR, RS, SS, DD, LB, FA, CE, MH, SU

**B方案打分补充** (`scripts_v2/score_domains.py`):

```python
score = 0.4 * normalize(cross_ref_count)   # VARIABLE_INDEX 引用次数
      + 0.4 * normalize(examples_tokens)    # examples.md 文件 token 数
      + 0.2 * (20 if domain in T_test_hits else 0)  # T1-T8 涉及域 +20

T_test_hits = {AE, PC, PP, EX, MH, LB}  # 从 v1 test_results.md 抽
```

取 top 5-8 域, 与用户清单合并去重, **预计补 5-8 域** (AE/PC/PP/VS/EG/QS/IE 等).

**最终高频域名单**: 25-28 域 (用户 20 + 打分新增 5-8)

**产物**: `output_v2/10_examples_data_high.md` (软目标 ≤30K, 硬上限 50K)

**算法细节**:
- 解析每域 examples.md, 仅保留 markdown table 段
- Variable Name | Value 双列, 去除冗长 description 叙述
- 保留 Example 标题 + 关键 Description (≤2 行)
- 保留共享 examples 链接 (EX/EC, MB/MS 等)

### 2.5 批 4: examples 剩余域数据表 (+80-150K)

**范围**: 63 域中除批 3 已覆盖的 25-28 域之外的 ~35 域。

**算法**: 同批 3, 但对 examples 内容更长的域 (>20K tokens) 启用"采样压缩":
- 每域取前 3 个 Example 完整 + 后续 Example 仅保留标题
- 共享 examples 仍保留链接

**产物**:
- 主目标: `output_v2/10b_examples_data_others.md` (≤50K)
- 若超 50K 硬上限: 按 SDTM Class 拆分为 `10b_findings.md` + `10c_events_interventions.md`

**v2.4 文件数**: 13 (若不拆) 或 14 (若拆 1 次), 仍在软目标 ≤15

### 2.6 批 5: terminology 高频 codelist Term 值 (+200-400K)

**圈定算法** (`scripts_v2/score_codelists.py`):

```python
score = 0.5 * normalize(domain_ref_count)  # 被多少个 domain 的 spec.md 引用
      + 0.3 * normalize(term_count)         # codelist 内 Term 数 (中等优先, 太大太小都减分)
      + 0.2 * (50 if codelist in T6_T7_T8 else 0)  # 测试题涉及的 codelist 加成
```

取 top 200 codelist, 覆盖 terminology 91 个文件中预计 ~50% 的 token 量 (因为高频 codelist 内 Term 值多)。

**算法细节**:
- 完整保留 codelist 标题 + 描述 + 全部 Term 值
- Term 表保留 5 列: Code | Term | Synonyms | Definition (截断 ≤200 字符) | Related Domain
- 不展开 NCI Concept Description 长文本 (链接到 NCI 即可)

**产物**:
- 主目标: `output_v2/11_terminology_high.md` (≤200K, 硬上限 250K)
- 若超 250K: 按 terminology 三大子目录拆 `11a_core.md` + `11b_questionnaires.md` + `11c_supp.md`

**v2.5 文件数**: 14 或 16 (拆 3 次), 接近软目标上限 15

### 2.7 批 6: terminology 中频 codelist Term 值 (+300-500K)

**范围**: 批 5 之外的 codelist 中, 打分前 200-500 名的 ~300 个 codelist。

**算法**: 与批 5 一致, 但对超大 codelist (Term > 100) 启用 "采样":
- 完整保留 Code + Term + Definition (截断 ≤100 字符)
- 不保留 Synonyms 大段
- 不保留 NCI Description 链接 (减少元数据膨胀)

**产物**:
- 主目标: `output_v2/12_terminology_mid.md` (≤500K, 硬上限 600K)
- 必拆: `12a_terminology_mid_core.md` + `12b_terminology_mid_questionnaires.md` (各 ~250K)

**v2.6 文件数**: 16-17, 触及软目标上限 15 → 接受溢出, 但低于硬上限 20

## 5. 验收标准

### 5.1 Layer 1: 自动检查 (脚本层, 每批后)

| # | 检查项 | 标准 | 适用批 |
|---|-------|------|-------|

## 8. 中庸策略的"提前终止"协议

设计明确允许在任意阶段提前终止, 视当前为 v2 终态。提前终止的合理触发:

| 触发 | 处理 |
|------|------|
| RAG 衰减 ≥2 题 | 立即停, 进收尾 |
| 用户决定 (疲劳/其他优先级) | 主控接受, 写 RETROSPECTIVE_V2 时记录"主动终止于 v2.X" |
| 当批新题 PASS = 0 且重做仍 0 | 主控分析: 是 RAG 容量瓶颈还是内容选错; 决策跳批或终止 |
| 累计 token 超 1.5M | 不允许进入下批, 视为 v2 终态 |

**v2.3 之后任何阶段的中间态都是有价值的成果**: 用户可在 v2.3/v2.4/v2.5 任一停下并视为完结。
