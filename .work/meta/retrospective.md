# 阶段性反思：Issue 1 & Issue 2 根因分析与预防机制

> 创建日期: 2026-04-15
> 适用范围: 本项目全阶段 + 同类 AI 辅助知识库项目

---

## 1. 两个 Issue 的共性根因

Issue 1（图像页码偏移）和 Issue 2（内容空洞过验证）表面不同，但共享同一条因果链：

```
AI 能力边界被高估 → 流程设计没有为 AI 弱项设置补偿机制 → 产出中的缺陷被当作正确结果固化
```

| 维度 | Issue 1: 页码偏移 | Issue 2: 内容空洞 |
|------|-------------------|-------------------|
| **AI 被信任做什么** | 视觉读取 PDF 精确定位页码 | 修复内容缺失并自行判定完成 |
| **AI 实际做到什么** | 只能估算（±2~4 页） | 只能重建骨架 + 打占位标记 |
| **为什么没被拦住** | 无程序化页码校验机制 | PASS 标准允许"标记了缺失"通过 |
| **谁发现的** | 用户人工抽查 | 用户人工对照 PDF |

---

## 2. 三个前置条件（缺一不可）

### 2.1 用定性标准代替定量标准

Issue 2 的 PASS 判定是"结构正确 + 已有内容准确 + 缺失已标记"——回答的是"看起来对不对"，而不是"覆盖了多少"。

如果当时有行数比值（md 行数 / PDF 页数），340 行 ÷ 38 页 ≈ 9 行/页，就能立刻发现异常。修复后的 ch04 是 654 行 / 38 页 ≈ 17 行/页。

### 2.2 自写自判（same-context self-validation）

两个 issue 的验证/修复都在同一个 agent 上下文中完成：
- 写的 agent 知道自己标记了占位，但认为"标记了就算处理了"
- 判的 agent 看到结构完整、标记清晰，就给了 PASS
- 本质上是让考生给自己批卷

后来修复计划中引入的"写/审分离 + 独立 subagent 复核"正是针对这个问题，实践证明有效（8/8 节通过独立复核）。

### 2.3 把 AI 的估算值当成事实值

Issue 1 中，Sonnet agent 标注的页码本质是"估算值"——Read tool 返回内容中页码边界不清晰，agent 只能推断。但流程中没有区分"精确值"和"估算值"，全部作为权威数据写入了 page_index.json 和 image_inventory.md。

## 3. AI 能力边界认知

| AI 擅长 | AI 不擅长 |
|---------|-----------|
| 结构化数据转换（xlsx → md） | 精确定位 PDF 页码 |
| 内容比对（找差异） | 判断"够不够完整" |
| 格式校验（标记、节号） | 衡量覆盖率 |
| 批量处理重复模式 | 评估自己的产出质量 |

---

## 4. 四条预防规则

### 规则 1：定量 PASS 标准（必须）

```
覆盖率 = md 已覆盖要点数 / PDF 要点总数
PASS 条件:
  - 覆盖率 ≥ 95%
  - 零占位标记（待补全/TODO/待后续/部分覆盖）
  - 行数/页数比值合理（本项目基准: ≈15-20 行/页）
```

### 规则 2：写/审必须分离（必须）

- 补写内容的 agent 和复核的 agent 不能共享上下文
- 复核 agent 必须独立读取 PDF + md，输出结构化覆盖率报告
- 已在 Issue 2 修复中验证有效

### 规则 3：AI 估算值必须标记（必须）

- 凡是 AI 无法程序化确认的数据（如 PDF 页码），标注 `(estimated)` 或 `(±N)`
- 不允许将估算值直接写入权威索引
- 需要精确值时，用程序化方法（逐页读取 + 关键词匹配）或人工核实

### 规则 4：人工抽样检查点（强烈建议）

- 每完成一个阶段，由用户随机抽取 3-5 个样本对照 PDF
- Issue 1 和 Issue 2 都是用户抽查发现的——这是唯一有效的兜底机制
- 关键是把"用户抽查"从偶然行为变成流程中的正式步骤

---

## 5. 适用场景

以上规则适用于任何 **AI 辅助从非结构化源（PDF、扫描件、图像）提取结构化知识** 的项目。核心原则是：

> AI 擅长找差异，不擅长判断"够不够"；擅长处理结构，不擅长衡量完整性。
> 把量化标准、写审分离、人工抽查三个机制嵌入流程，就能在保留 AI 效率优势的同时堵住系统性漏洞。
