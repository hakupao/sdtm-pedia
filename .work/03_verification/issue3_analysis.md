### 1.2 发现方式

用户在 Step 4-3 人工抽查阶段，从 §4.1.9 开始顺序阅读 md 文件并对照 PDF。发现：
- §4.1.9 修复后仍然只覆盖了 PDF 前半段，后半段（SPDEVID 扩展键、phalanges 5 要素、SGBESCR 示例）完全缺失

- §4.3.7 完全不存在——既没有在终验中被发现，也没有在修复中被添加

用户观察到一个明显趋势：**前面的节精度尚可，越到后面错误越多、缺失越严重**。到 §4.4 时，缺失密度已经高到不值得继续逐节对照，因此用户终止了人工复查。

**结论**: 用户未完成复查不代表后面没问题，恰恰相反——后面很可能问题更多。因此本次修复范围是 **ch04 全部内容（§4.1–§4.5）**，不限于用户已检查的节。

### 1.3 关键信号

用户在打开文件时，直觉性地注意到 §4.4 之前的多个节"太短了"，这与之前 Issue 2 的发现模式完全一致——**人工阅读可以感知"密度不对"，AI 验证难以感知**。

## 3. 根因分析：为什么三轮验证都没查出来

### 3.1 时间线上的三次"漏网"

| 轮次 | 方法 | 为什么没查到 |
|------|------|------------|
| **Issue 2 修复**（Step 3 阶段） | 修复有占位标记的节 | §4.2.3、§4.2.6、§4.3.7 从未被标记为"待补全"——它们有内容，只是内容太少。占位标记扫描只找空洞节，不找"太短的节" |
| **Step 4 终验**（5 个 agent） | 独立 agent 读 PDF+md 比对 | 终验 agent 对 §4.1 的覆盖率报 66.7%，确实发现了很多问题。但 agent-final-ch04-front 的 PDF 阅读范围是 p.22-40，实际 §4.1 只到 p.26。**agent 读了 p.26-40 的内容（§4.2-§4.3 区域）但没有系统性地逐节比对这些节**——它只聚焦于 §4.1 范围内的问题 |
| **Step 4 修复+复核** | repair agent 修已知问题 → reviewer agent 抽查修复节 | repair agent 只修终验报告中列出的具体 issue。reviewer agent 只复核被修改的节。**没有任何 agent 对未被报出的节做过覆盖率检查** |


### 3.3 核心发现

**Step 4 终验的 agent 拆分方案是按"页码均分"（~19页/agent）而非按"章节边界"拆分的。**

这导致：
- §4.2 全部内容（p.28-36）和 §4.3 前半部分（p.36-40）落入 front agent 的 PDF 阅读范围，但 front agent 的任务只是检查 §4.1
- back agent 从 p.41 开始读，完全跳过了 §4.2 和 §4.3 的大部分
- **§4.2.3、§4.2.6、§4.3.7 从未被任何 agent 正式验证过**

### 3.4 为什么人工复查能发现

| 维度 | AI 终验 | 人工复查 |
|------|--------|---------|
| 阅读方式 | 按预设页码范围读 PDF，按 prompt 中的节号检查 | 顺序阅读 md，直觉感知密度异常 |
| 覆盖范围 | 受 prompt 约束，只检查被分配的节 | 自由浏览，不受预设范围限制 |
| 异常检测 | 只能发现"PDF 有 md 无"的内容差异 | 能感知"这节太短了"的密度异常 |
| 盲区 | 完全依赖 prompt 正确分配覆盖范围 | 无盲区（但速度慢、精力有限） |

## 5. 方案 A 解决方案：ch04 逐节全面重提取

### 5.1 核心原则

基于 Issue 2 和 Issue 3 的教训，本次修复必须：

1. **按章节边界拆分 agent，不按页码均分** — 每个 agent 负责一个完整的顶级节（§4.1 / §4.2 / §4.3 / §4.4 / §4.5），读取该节的确切 PDF 页码范围
2. **全文通读，不是只修已知问题** — agent 的 prompt 是 "extract all content from this section"，不是 "fix these issues"
3. **强制输出逐子节覆盖率** — 输出必须包含每个子节（如 §4.2.3、§4.3.7）的独立覆盖率行
4. **写/审分离 + 独立复核** — 提取 agent ≠ 复核 agent
5. **人工终检** — 修复后再次提交用户抽查

### 5.2 前置步骤：确定每节的精确 PDF 页码范围

在启动任何 agent 之前，先逐节确认精确的 PDF 页码边界（不能靠 agent 估算）：

| 顶级节 | 确认后 PDF 页码 | 状态 |
|--------|---------------|------|
| §4.1 General Domain Assumptions | p.22–27 (§4.1.9 表格在 p.27 顶部结束) | ✅ 已确认 |
| §4.2 General Variable Assumptions | p.27–36 (p.27 中部开始，§4.2.9 在 p.36) | ✅ 已确认 |
| §4.3 Coding and CT Assumptions | p.36–39 (p.36 中部开始，§4.3.7 在 p.38-39，§4.4 从 p.39 中部开始) | ✅ 已确认 |
| §4.4 Actual and Relative Time | p.39–51 (p.39 中部开始，§4.4.11 在 p.48-51，"Linking and Disease Milestones" 结束于 p.51) | ✅ 已确认 |
| §4.5 Other Assumptions | p.51–59 (p.51 中部开始，§4.5.9 Baseline Values 在 p.59 结束) | ✅ 已确认 |

### 5.3 Agent 拆分方案

| Agent | 负责节 | PDF 页码 | 预估页数 | 任务类型 |
|-------|--------|---------|---------|---------|
| extract-4.1 | §4.1（全部子节） | p.22–27 | 6 | **全面审查** — §4.1.9 后半段已确认缺失，其余子节也需逐一验证完整性 |
| extract-4.2 | §4.2（全部子节） | p.27–36 | 10 | **全面重提取** — §4.2.3、§4.2.6 已确认严重不足，其余子节逐一审查 |
| extract-4.3 | §4.3（全部子节） | p.36–39 | 4 | **全面重提取** — §4.3.7 已确认完全丢失，§4.3.1-§4.3.6 逐一审查 |
| extract-4.4 | §4.4（全部子节） | p.39–51 | 13 | **全面重提取** — §4.4.1/4.4.2 已确认严重压缩；§4.4 引言缺失；后段用户未复查但推断问题更多 |
| extract-4.5 | §4.5（全部子节） | p.51–59 | 9 | **全面审查** — 用户未复查此节，不可假设无问题，需逐子节比对 |

**关键改进**: 每个 agent 只负责一个顶级节，且 PDF 页码范围精确对应该节内容，不存在"读到了但不检查"的盲区。

### 5.4 Agent Prompt 模板（区别于之前的终验 prompt）

```
你是内容提取 agent（不是验证 agent）。

任务: 确保 md 文件的 §X.Y 节完整覆盖 PDF 对应页码的全部内容。

方法:
1. 读 PDF 指定页码范围
2. 列出该节的全部子节编号和标题
3. 对每个子节:
   a. 列出 PDF 中的全部要点（定义、规则、示例、表格、交叉引用）
   b. 读 md 对应子节
   c. 逐要点比对，标记 ✅已有 / ❌缺失
   d. 对 ❌缺失的要点，直接用 Edit 工具补写到 md
4. 输出逐子节覆盖率报告

强制要求:
- 必须逐子节报告，不允许合并为一个总覆盖率
- 每个子节必须有独立的 PASS/FAIL 判定
- 完全缺失的子节（如 §4.3.7）必须先创建再填充
- 不得跳过任何子节
```

### 5.5 复核方案

每个 extract agent 完成后，启动对应的独立 review agent：
- review agent 读同一段 PDF + 修复后的 md
- **只检查覆盖率和准确性**（不做修复）
- 输出逐子节 PASS/FAIL
- 如有 FAIL，回到 extract agent 补充，再启新的 review agent

### 5.6 §4.4 的特殊处理

§4.4 有 17 页 PDF，是单 agent 的上限。需要拆分：

| Sub-agent | 覆盖节 | PDF 页码 | 页数 |
|-----------|--------|---------|------|
| extract-4.4a | §4.4 引言 + §4.4.1 + §4.4.2 + §4.4.3 | p.39-43 | 5 |
| extract-4.4b | §4.4.4 + §4.4.5 + §4.4.6 + §4.4.7 + §4.4.8 | p.43-47 | 5 |
| extract-4.4c | §4.4.9 + §4.4.10 + §4.4.11 | p.47-51 | 5 |

**注意**: 这次的拆分是按子节边界拆，每个 sub-agent 负责连续的子节，PDF 页码范围精确覆盖这些子节，不存在盲区。

## 6. 经验积累：Issue 3 带来的新规则

### 规则 5：Agent 拆分必须按章节边界，不按页码均分（新增）

```
错误做法: ch04 拆成 p.22-40 / p.41-59（按页码中点切）
正确做法: ch04 拆成 §4.1 / §4.2 / §4.3 / §4.4 / §4.5（按章节切）

原因:
- 页码均分会导致"agent A 读到了 §X 的内容但不在其任务范围内"
- 章节边界拆分确保每个 agent 的阅读范围 = 检查范围 = 修复范围
- 如果单章节 PDF 页数超过 agent 处理上限（约 20 页），按子节边界拆分
```

### 规则 6：终验/提取 agent 的 prompt 必须包含完整子节清单（新增）

```
错误做法: "验证 §4.2-4.5 的内容"（让 agent 自己发现子节）
正确做法: "验证以下子节: §4.2.1, §4.2.2, §4.2.3, §4.2.4, §4.2.5, §4.2.6, §4.2.7, §4.2.8, §4.2.9, §4.3.1, §4.3.2, ..."

原因:
- 如果 agent 的 PDF 阅读范围不完整（如从 p.41 开始跳过了 §4.2），它不知道 §4.2 存在
- 在 prompt 中明确列出所有子节编号，agent 就会主动寻找每个子节
- 这实质上是给 agent 一个 checklist，而不是依赖它自己构建 checklist
```

### 规则 7：修复 agent 的任务必须是"确保完整覆盖"，不是"修复 N 个已知问题"（新增）

```
错误做法: "修复以下 8 个问题: Issue 1: ..., Issue 2: ..."
正确做法: "确保 §4.2 的全部内容完整覆盖 PDF p.28-36，修复一切缺失"

原因:
- "修复已知问题" 天然只能处理已发现的缺陷
- "确保完整覆盖" 要求 agent 自己发现并修复所有缺失（包括未被前序步骤发现的）
- 这消除了对前序验证完整性的依赖
```

### 与 retrospective 四条规则的关系

| 规则 | 状态 | Issue 3 验证 |
|------|------|-------------|
| 规则 1: 定量 PASS 标准 | 仍然有效 | 终验确实计算了覆盖率，但覆盖范围有盲区 |
| 规则 2: 写/审分离 | 仍然有效 | 复核 agent 确实独立执行，但只复核被修复的节 |
| 规则 3: AI 估算值标记 | 仍然有效 | 本次不涉及估算值问题 |

### 7.4 Evidence 保存规范

| Evidence 类型 | 保存方式 | 示例 |
