# Cloudflare Pages settings (sdtm-pedia)

> Canonical settings for the Cloudflare Pages project that serves this site.
> Source: deploy by **manual GitHub auto-connect** in the CF dashboard
> (no Wrangler / no MCP automation). Daisy logs into Cloudflare, picks Pages,
> selects the GitHub repo, and CF builds on every push to `main`.

## Project basics

| Setting | Value |
|---|---|
| Project name | `sdtm-pedia` |
| Production branch | `main` |
| Repository | `hakupao/sdtm-pedia` (auto-connected via GitHub OAuth in CF dashboard) |
| Custom domain | none (defer to v1.1; the assigned `*.pages.dev` URL is the public URL for now) |

## Build configuration (CF dashboard → Settings → Builds & deployments)

| Field | Value |
|---|---|
| Framework preset | `Astro` (CF auto-detects; if not, set to `None` and fill manually) |
| Build command | `cd web && npm install && npm run build` |
| Build output directory | `web/dist` |
| Root directory (advanced) | `/` (leave default) |
| Build system version | `2` (CF Pages V2 build system; default for new projects) |

## Environment variables (CF dashboard → Settings → Environment variables)

Add these to **both** "Production" and "Preview" scopes:

| Variable | Value | Why |
|---|---|---|
| `NODE_VERSION` | `22.12.0` | Override CF Pages default. `web/package.json` `engines.node` requires `>=22.12.0` (Astro 6 + React 19 baseline). The repo also has a root `.nvmrc` so this is belt-and-suspenders. |
| `PUBLIC_RELEASE_PUBLISHED` | `true` | Toggles the DownloadsSection bundle grid. Default-on in code; explicit here for transparency. Set to `false` only during a release-prep window where `web/src/data/downloads.json` references a tag not yet published to GitHub Releases. |

## Build watch paths (CF dashboard → Settings → Builds & deployments → Build watch paths)

Limit which path changes trigger a rebuild (otherwise every commit to `.work/` etc. wastes a build minute):

**Include**:
- `web/**`
- `ai_platforms/release/v1.0/**` (the content collection's `glob` source — guide markdown changes must rebuild)

**Exclude** (implicit when Include is set, but listed for clarity): `.work/**`, `source/**`, `knowledge_base/**`, `docs/**`, `.omc/**`.

## First-deploy walkthrough (one-time)

1. Log into Cloudflare → **Workers & Pages** → **Create application** → **Pages** → **Connect to Git**.
2. Authorize the GitHub app (if first time), pick `hakupao/sdtm-pedia`.
3. **Project name**: `sdtm-pedia`. **Production branch**: `main`. → Continue.
4. **Build settings** screen — fill the table above (Build command + Output dir + Root dir).
5. Expand **Environment variables** — add `NODE_VERSION=22.12.0` and `PUBLIC_RELEASE_PUBLISHED=true`. → Save and Deploy.
6. First build runs ~3-5 minutes. CF dashboard shows live log. On success → green check + assigned `https://sdtm-pedia.pages.dev` URL.
7. Open `https://sdtm-pedia.pages.dev/zh/`. Verify:
   - Landing page renders all 5 sections (Hero / Platforms / Compare preview / Demo / Downloads).
   - Lang switcher (top-right) flips to `/en/` and `/ja/` cleanly.
   - Click "READ THE GUIDE" → lands on `/zh/guide/user-guide` with sidebar + TOC + prev/next.
   - DownloadsSection 4-bundle grid links resolve to `https://github.com/hakupao/sdtm-pedia/releases/download/v1.0/...` (already published).
8. **Configure build watch paths** (after first successful build): Settings → Builds & deployments → Build watch paths → enter the Include list above → Save.

## Preview deployments (automatic)

After the project is set up, CF Pages will auto-build:

- Any push to a **non-`main` branch** → preview URL `https://<branch-slug>.sdtm-pedia.pages.dev`.
- Any **pull request** → preview URL on the PR check.

So Phase 6/7 work can be developed on a feature branch (e.g. `phase-6-compare`) and reviewed at a preview URL before merging to `main` and going live.

## Promotion to production

The first push to `main` after CF Pages connection IS the first production deploy. There is no separate "promote" button — `main` = production, full stop. To roll back, use CF dashboard → Deployments → pick a previous successful deploy → "Rollback to this deployment".

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| Build fails: `npm ERR! Unsupported engine` or `code EBADENGINE` | NODE_VERSION env var unset or = 18/20 | Confirm `NODE_VERSION=22.12.0` in env vars (Production AND Preview scopes); rebuild |
| Build succeeds but landing 404s | Build output dir wrong | Confirm `web/dist` (NOT `dist` or `web/build`) |
| Build succeeds but DownloadsSection shows "release-pending" placeholder | `PUBLIC_RELEASE_PUBLISHED` set to `false` (or unset) | Set to `true` in env vars; rebuild |
| Build runs on every `.work/` commit | Build watch paths not configured | Set Include paths per the section above |
| `pagefind` not found during build | CF pruning devDependencies | Add `NPM_CONFIG_PRODUCTION=false` to env vars (forces install of devDependencies). Should not be needed by default — Pages installs devDependencies during build. |
