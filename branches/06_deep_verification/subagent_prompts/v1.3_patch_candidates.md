## R10 (already applied to batch 11 prompts) — NEW from O-P1-25

**Rule**: Spec table verbatim is literal, including PDF rendering artifacts. Specifically: when a spec table cell wraps text and the PDF cell renders no space at the line break (e.g. `domains.This may be...` — no space after period because cell wrap dropped it), atom verbatim MUST preserve `domains.This` literally. **DO NOT insert well-meaning whitespace** to "fix" sentence boundaries. ## R11 (NEW from batch 11 O-P1-31) — TABLE_ROW trailing empty cell preservation

**Rule**: TABLE_ROW verbatim must preserve **exactly N PDF columns** including each trailing empty cell as its own `| |` representation. **DO NOT collapse** consecutive trailing empty cells into a single empty representation. If PDF table has 10 columns and a row populates 8 with 2 trailing empties, atom verbatim must end with `| | |` (3 final pipes for 2 empty cells), NOT `| |` (2 final pipes for 1 collapsed empty representation).

**Test case**: batch 11 ig34_p0102_a022 ZOLOFT + a023 PROZAC Example 3 cm.xpt row — PDF has 10 columns (Row | STUDYID | DOMAIN | USUBJID | CMSEQ | CMTRT | CMPRESP | CMOCCUR | CMSTAT | CMREASND), Row 1/2 populate 8 + 2 trailing empties (CMSTAT empty + CMREASND empty). Writer rendered `... | Y | Y | |` (1 collapsed empty representation) instead of `... | Y | Y | | |` (2 separate empty representations).

**Verification methodology**: count internal pipe characters in atom verbatim. For an N-column TABLE_ROW with all cells populated → exactly N-1 internal pipes (or N if outer-pipe style). Each empty cell adds one extra `|` per missing value. Cross-check against TABLE_HEADER atom column count for the same table.

**Recommended writer self-validation step**: after composing TABLE_ROW verbatim, count `|` characters and compare to TABLE_HEADER for same table. If mismatch, recompose preserving each empty cell.

## R12 (NEW from batch 11 O-P1-28) — Transition page full-content discipline

**Rule**: Transition pages (PDF page contains 2+ section heading boundaries, e.g. `§X.Y` ending and `§X.(Y+1)` starting on same page) must atomize ALL physical content on the page, including the **previous section's tail content** before processing the next section. DO NOT skip CM tail content just because the page also contains §6.1.3 heading. The page's physical content is N atoms, all N must be captured regardless of where section transition occurs.

**Test case**: batch 11 11a writer p.103 = under-extraction 6 atoms vs PDF expected ~21. Page 103 spans CM Example 5 narrative tail (5 list items + cm.xpt label + TABLE_HEADER + 6 TABLE_ROWs = 13 atoms under §6.1.2 CM) → §6.1.3 Exposure Domains heading → §6.1.3 intro (1 paragraph 2 sentences = 2 atoms under §6.1.3). 11a writer captured only 3 TABLE_ROWs from previous page's table continuation (incomplete) + §6.1.3 heading + 2 (collapsed-into-1) intro SENTENCEs, dropping all CM Example 5 + cm.xpt content.

**Recommended writer protocol**:
1. Read PDF page once, identify ALL physical content top-to-bottom.
2. Identify section boundaries: every `§X.Y` heading line is a transition.
3. Each region between transitions belongs to its own parent_section.
4. Atomize each region completely BEFORE moving to next region.
5. Self-validation: density check — if your atom count for a dense page is under 10, sanity-check by re-reading the page; spec/example tables on page should yield 1 TABLE_HEADER + N TABLE_ROWs for an N-row table.

## R13 (NEW from batch 11 O-P1-30) — Numbered list item discipline regardless of formatting

**Rule**: If PDF prints `<num>. <text>` and `<num>` matches the surrounding numbered list (e.g. list goes 1, 2, 3, 4 and item 4 is `4. Variables for timing relative to a time point`), the atom is **LIST_ITEM** regardless of the item's bold/italic/standalone-line formatting. Bold formatting alone does NOT promote a numbered list item to HEADING.

**Disambiguation rules**:
- HEADING: section number matches TOC (e.g. `6.1.3.1 Exposure (EX)` is in TOC → HEADING) OR is a new sub-section starting a new content block with formal section formatting (bold + larger font + standalone line + matches a TOC entry).
- LIST_ITEM: bold or italic formatting on a numbered/lettered list item that continues a list pattern (e.g. surrounding items are `1.`/`2.`/`3.` and you see `4.` with bold formatting — still a list item).
- When in doubt: if it appears in the PDF Table of Contents, it's HEADING; if it doesn't, it's LIST_ITEM (or SENTENCE for non-numbered text).

**Test case**: batch 11 11a a006 "4. Variables for timing relative to a time point" + a013 "5. Although any identifier variables..." — both numbered list items 4 and 5 of CM Assumptions list, NOT in TOC. Writer mis-classified as HEADING due to bold formatting + standalone line layout.

## R14 (NEW from batch 11 O-P1-27) — Writer DONE atoms=N self-validation

**Rule**: Before reporting `DONE atoms=N`, writer MUST count actual lines in `output_file` and ensure N matches. If internal counter (running tally during atomization) disagrees with file line count, recompute from file (`wc -l output_file` equivalent). Report file ground truth, not internal counter.

**Test case**: batch 11 11a writer reported `DONE atoms=131` but actual file had 122 lines (-9). Same writer-family pattern as 09b (139 reported / 120 actual). Executor pattern (10a/10b/11b) shows 100% strict match.

**Recommended writer protocol**: at end of batch, before composing DONE message:
1. Count lines in output_file (or count Write tool calls if file-IO based).
2. Compare to internal counter.
3. If mismatch, recompute: `actual_atoms = wc -l output_file`.
4. Use `actual_atoms` value in DONE message.

This is a writer-family-specific patch — executor has different internal counting that does NOT exhibit this drift.

## O-P1-26 (formal v1.3 codification) — TABLE_ROW outer-pipe convention

**Rule**: All TABLE_HEADER + TABLE_ROW atom verbatim MUST use **outer-pipe style**: `| field1 | field2 | ... | fieldN |` with leading `|` and trailing `|` plus internal `|` separators. DO NOT use no-leading-pipe style `field1 | field2 | ... | fieldN`.

**Rationale**: outer-pipe style makes column count unambiguous (N+1 pipes for N columns); makes empty cells visually consistent (`| |` for any empty position including leading/trailing); aligns with markdown table syntax.

**Test case**: batches 09-11 have mixed styles. Rerun executor (batch 11 p.103) uses outer-pipe `| Row | ... |`; original 11a writer (batch 11 p.101/p.102) uses no-leading-pipe `Row | STUDYID | ...`; 11b executor (batch 11 p.108) uses no-leading-pipe. **Mixed within batch** = O-P1-26 reaffirmed.

**Migration impact**: NOT a backward-compatible change for prior batches — would invalidate ~100 TABLE_ROW atoms in batches 02-11. Recommend defer enforcement to v1.3 (start batch 12) without retroactive root-file rewrite (record as "convention adopted at batch 12, prior atoms retain inconsistent style as historical artifact").

### NEW v1.3 patch candidates surfaced post-batch-15

#### NEW1 — Drift cal dual-threshold (NEW from batch 15 lesson)

**Rule**: Drift calibration MUST evaluate BOTH `strict_count_match ≥80%` AND `verbatim_hash_overlap ≥80%`. If either threshold fails, investigate. Strict count alone hides typo-class corruption (writer-family STUDIID×8 + supple→suppbe + cross-domain TABLE_HEADER pollution + column-name typos) when atom_id structures match exactly but content has systemic verbatim divergence.

**Test case**: batch 15 p.147 drift cal — strict 97.4% (38/39) PASS would have masked 59% verbatim divergence; only verbatim hash overlap 41% LOW surfaced the deep root cause analysis revealing 4 HIGH bugs (STUDIID/supple/bs swap/relspec/relrec/p.146 under-extraction).

**Implementation**: drift cal script computes `set(verbatim_hash for atom in baseline) ∩ set(verbatim_hash for atom in rerun) / max(|baseline|, |rerun|)`; threshold ≥80%. Both thresholds reported.

#### NEW2 — Writer spec-table self-validation (NEW from batch 14 O-P1-36)

**Rule**: Writer SHOULD perform character-level self-validation step on spec table cells before DONE: re-read each variable name + Core column + CT column from the verbatim atom and compare to PDF rendering. Catch typos (AERLPRT/AELLT/AELLTCD multi-char drift) early. Particularly applies to writer-family agents (executor-family less affected).

**Test case**: batch 14 14b on p.136-140 writer 14b R7+R14 strict count match held but R10 verbatim FAIL across all 5 pages — distinct failure mode where writer atomized correct count but corrupted variable names (AEUNANT→AEURANT etc).

**Recommendation**: writer prompt explicitly enumerate spell-check step on every spec TABLE_ROW.

#### NEW3 — Option E rerun explicit outer-pipe + null-key convention (NEW from batch 14 O-P1-37)

**Rule**: Option E executor rerun prompt MUST require: (a) outer-pipe `| field1 | field2 | ... | fieldN |` for TABLE_ROW + TABLE_HEADER + (b) explicit `heading_level: null` + `sibling_index: null` keys for non-HEADING atoms (matching standard executor 14a output convention).

**Test case**: batch 14 Option E rerun output 14b 72 atoms drifted convention (24 inner-pipe + 67 omitted-key) vs 14a (52 outer-pipe + 95 null-key). Bulk Option H fix 91 atoms required to restore within-batch consistency.

#### NEW4 — Dataset filename HEADING vs CODE_LITERAL convention codification (NEW from batch 13 O-P1-36)

**Rule**: Either (a) STRICT — `*.xpt` / `*.sas7bdat` / `*.csv` dataset filenames are ALWAYS CODE_LITERAL per R9 (deprecate caption-style HEADING for dataset filenames), OR (b) EXPLICIT EXCEPTION — caption-context exception MAY HEADING when dataset filename appears immediately before a data table as visual caption (with both interpretations allowed).

**Test case**: batch 13 writer-family 13a treated ml.xpt as HEADING L5 sib=None caption-style; executor 13b treated pr.xpt × 3 + eg.xpt + relrec.xpt as CODE_LITERAL. Same-batch convention drift caused inconsistent atomization. Recommendation: (a) STRICT preferred for downstream parser determinism.

#### NEW5 — R12 chapter-level transition discipline strengthening (NEW from batch 14 p.133 success case)

**Rule**: When PDF page contains chapter-level transition (e.g. previous-chapter tail content + new chapter heading + new sub-section heading on same page), writer MUST partition page into 3 zones: (Zone 1) tail content under previous parent_section; (Zone 2) new chapter HEADING + intro under §X parent; (Zone 3) new sub-section HEADING + sub-content under §X.Y parent. Each zone's parent_section is calculated BEFORE atom emission.

**Test case**: batch 14 p.133 successfully partitioned (Zone 1 §6.1.6 SU tail + Zone 2 §6.2 chapter NEW intro + Zone 3 §6.2.1 AE NEW). batch 15 p.143 + p.148 also partitioned successfully (BE chapter start at §6.2.2; CE chapter start at §6.2.3). Pattern reproducible.
